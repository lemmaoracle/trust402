## Why

Trust402 provides buyer-side ZK proof protocols (`@trust402/protocol`, `@trust402/identity`, `@trust402/roles`) but lacks an interactive demo that showcases the end-to-end flow. External developers and stakeholders need a CLI experience that demonstrates the behavior of an AI agent using ZK proofs to auto-pay for a paid API, verify the received data via Lemma proofs, and present a unified summary.

## What Changes

- Add `packages/demo` as a new pnpm workspace package with four sub-packages:
  - `packages/demo/resource` — x402 paid REST API server using `@lemmaoracle/x402` (Lemma-augmented x402). Serves corporate IR financial data (e.g. `GET /ir/2026q1`) behind a $0.01 USDC paywall. The response includes an `attestation` field (docHash) that links to a Lemma proof verifying the financial data's authenticity
  - `packages/demo/agent` — Demo CLI that simulates an AI agent. Receives a fixed user query → infers a paid API is needed → prompts for IdentityArtifact generation if absent → executes payment via `wrapFetchWithProof` → verifies the attestation via Lemma API → outputs proof summary to stdout
  - `packages/demo/normalize` — Minimal Rust normalization module compiled to WASM, used to transform raw financial data into the circuit input format (simplified version of `lemma/packages/agent/normalize`)
  - `packages/demo/circuit` — Minimal ZK circuit (Foundry project) for verifying financial data attestations (simplified version of `lemma/packages/agent/circuits`)
- The demo agent uses a scripted reasoning flow (no real LLM — cost/latency/stability trade-off)
- No `DEMO_MODE` flag (always uses real network services)

## Capabilities

### New Capabilities

- `demo-resource`: x402 paid REST API resource server using `@lemmaoracle/x402`. Provides corporate IR endpoints (e.g. `/ir/2026q1`) that return financial data with a `docHash` attestation verifiable via the Lemma oracle
- `demo-agent`: CLI demo that simulates an AI agent. Loads SKILL.md → simulates reasoning → IdentityArtifact generation dialog → payment via `wrapFetchWithProof` → attestation verification via Lemma API → proof summary output
- `demo-normalize`: Minimal Rust→WASM normalization module for transforming financial data into ZK circuit input format. Pure normalization only — docHash is generated by the Lemma oracle, not by the normalizer
- `demo-circuit`: Minimal Foundry ZK circuit for verifying financial data attestations

### Modified Capabilities

(none)

## Impact

- New packages: `packages/demo/{resource,agent,normalize,circuit}`
- Dependencies: `@trust402/protocol`, `@trust402/identity`, `@trust402/roles`, `@lemmaoracle/x402`, `@lemmaoracle/sdk`, `@x402/fetch`, `@coinbase/x402`, `hono`, `chalk`, `ora`, plus Rust/Foundry toolchain for normalize and circuit
- No impact on existing packages (pure addition)
- New CLI entry point: `demo-agent` binary
- Network prerequisites: Lemma API (Relay prover), x402 facilitator, Base Sepolia access
- Build prerequisites: Rust + wasm-pack, Foundry

## Non-goals

- Real LLM integration (out of scope due to cost, latency, and stability concerns)
- Production-grade ZK circuits or normalization logic (these are simplified demo-only versions)
- `DEMO_MODE` / offline mode (always uses real network services)
- General-purpose agent framework or plugin architecture
- Full parity with `lemma/packages/agent` circuit and normalization complexity
