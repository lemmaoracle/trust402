## ADDED Requirements

### Requirement: Financial data normalization to field elements
The system SHALL expose a WASM-exported function that accepts a JSON string of financial data (reportId, company, period, revenue, profit) and returns an array of field elements suitable for ZK circuit input.

#### Scenario: Normalize Q1 2026 report
- **WHEN** the normalizer receives `{ reportId: "2026q1", company: "Example Corp", period: "2026-Q1", revenue: 1250000000, profit: 340000000 }`
- **THEN** it returns an array of field elements representing these values in the circuit's expected format

### Requirement: Rust source compiled to WASM
The normalization module SHALL be implemented in Rust and compiled to WASM via `wasm-pack`. The crate SHALL expose a `normalize` function callable from JavaScript.

#### Scenario: WASM build
- **WHEN** `wasm-pack build` is run in the `packages/demo/normalize` directory
- **THEN** a `pkg/` directory is produced with the WASM binary and JS bindings

### Requirement: Simplified normalization logic
The normalization SHALL use a minimal approach: convert each field to a fixed-size byte representation, then to a BigInt field element. No Poseidon hashing, no ECDSA operations — just straightforward field element conversion. The docHash is automatically generated by the Lemma oracle during document registration, so the normalizer does not need to compute it.

#### Scenario: Revenue field conversion
- **WHEN** the normalizer processes `revenue: 1250000000`
- **THEN** it converts the integer to a field element representation (e.g. `1250000000n` as a BigInt)

### Requirement: Deterministic output
The normalization SHALL be deterministic — the same JSON input always produces the same field element array. This ensures that the normalizer output used during document registration matches the output used during proof generation.

#### Scenario: Determinism
- **WHEN** the normalizer processes the same JSON input twice
- **THEN** both calls return identical field element arrays
